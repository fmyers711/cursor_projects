Key architectural change

This workflow represents a deliberate shift in how CHOA and ReliaQuest collaborate. ReliaQuest analysts work entirely within their own GreyMatter instance for L1 triage—they do not require direct access to CHOA's Splunk ES. Splunk ES remains CHOA's system of record for escalated investigation, correlation, and SOAR response. The integration focus moves from shared-console access to disciplined data flow into GreyMatter and escalation flow back into Splunk ES.

GreyMatter L1 triage engine — ReliaQuest analysts operate here exclusively for 24/7 monitoring, validation, and enrichment.
Splunk ES CHOA system of record — receives escalations, drives L2/L3 investigation, and maintains case history.
Splunk SOAR Automated response layer — triggered by Splunk ES escalations, CHOA-owned with human-in-the-loop gates.
Bidirectional API Connective tissue — shared case ID, standardized payloads, and closed-loop status sync.

Integration summary

  • GreyMatter = L1 triage engine — ReliaQuest analysts operate here exclusively.
  • Splunk ES = CHOA system of record — receives escalations and drives investigation.
  • Splunk SOAR = automated response layer — triggered by Splunk ES escalations, CHOA-owned.
  • Bidirectional API + shared case ID = the connective tissue enabling seamless, auditable handoffs.

1. Detection & data flow into GreyMatter

CHOA telemetry → Splunk ES → GreyMatter

  • CHOA telemetry (endpoint, network, identity, cloud, healthcare apps) is ingested into Splunk ES and normalized to CIM.
  • Relevant notables, risk-based alerts, and raw log context are forwarded from Splunk ES into GreyMatter via API/connector.
  • Map detections to MITRE ATT&CK so threats are described consistently across both platforms.
  • Ensure only in-scope data flows to GreyMatter to respect PHI handling and data-residency requirements.

Healthcare guardrail: PHI & data residency

Define a clean data contract before integration goes live. Specify exactly which telemetry fields, enrichment artifacts, and log sources are permitted to leave CHOA's environment. Escalation payloads should carry the minimum context ReliaQuest needs for triage—not full patient records or unnecessary clinical data. Document residency requirements and audit forwarding rules quarterly.

Data contract checklist

Element Source Destination Requirement
Notables & RBA alerts Splunk ES GreyMatter MITRE ATT&CK mapped; severity and rule metadata included
Raw log context Splunk ES GreyMatter Scoped to in-scope sources only; PHI fields redacted or excluded
Asset & identity context Splunk ES (CIM) GreyMatter Operational identifiers only; no unnecessary clinical attributes
Correlation ID Splunk ES notable GreyMatter case shared_case_id assigned at creation; immutable across lifecycle

2. Level 1 triage (within GreyMatter)

ReliaQuest L1 operating model

ReliaQuest analysts perform all monitoring, validation, and triage inside GreyMatter—no direct Splunk ES access required.

  • Apply standardized triage criteria: true/false positive, severity, and business-impact scoring.
  • Enrich alerts within GreyMatter using its native threat intel, asset, and user context.
  • Auto-close confirmed benign events in GreyMatter to prevent unnecessary escalation noise to CHOA.

Standardized triage criteria

Dimension ReliaQuest action in GreyMatter Outcome
True / false positive Validate against TI, asset context, and known-good baselines FP → auto-close; TP → continue triage
Severity Apply agreed severity matrix (critical / high / medium / low) Drives SLA timer and escalation threshold
Business impact Score clinical, operational, and PHI exposure risk Healthcare-specific weighting for escalation decisions
Enrichment completeness Attach TI, user, asset, and timeline context before decision Required fields in escalation payload if escalating

3. Escalation handoff (GreyMatter → Splunk ES)

Escalation without manual re-keying

  • Define agreed escalation thresholds (e.g., confirmed malware, PHI access anomalies, privileged account misuse).
  • On escalation, GreyMatter pushes the case into Splunk ES (or CHOA's ticketing system) via bidirectional API/webhook.
  • Standardize the escalation payload: original notable ID, ATT&CK mapping, GreyMatter enrichment, timeline, and recommended actions.
  • Preserve a shared case/correlation ID linking the GreyMatter case to the corresponding Splunk ES notable for chain-of-custody.
  • Attach SLA timers per severity so handoff accountability is measurable on both sides.

Escalation threshold examples

Scenario Escalate when Severity
Confirmed malware Active execution or persistence on endpoint; TI match confirms malicious Critical / High
PHI access anomaly Unusual volume, off-hours access, or role-inappropriate EHR queries Critical
Privileged account misuse Admin credential use from anomalous location, impossible travel, or lateral movement High
Phishing with credential capture User-reported or detected credential submission to known phishing infra High
Data exfiltration indicators Large egress to untrusted destination; DLP or network anomaly corroborated Critical

4. SOAR-driven response (Splunk SOAR, CHOA-owned)

Automated response triggered by Splunk ES escalation

Escalation into Splunk ES automatically triggers the relevant Splunk SOAR playbook. Use human-in-the-loop approval for high-impact actions affecting clinical or patient-care systems.

Suggested playbook actions by scenario

Scenario SOAR playbook actions Human-in-the-loop?
Phishing Detonate URLs/attachments, retrieve related emails, disable accounts Account disable on clinical staff → approval required
Endpoint compromise Isolate host via EDR, collect forensics, block IOCs Isolate clinical/system-critical host → approval required
Identity / IAM anomaly Force credential reset, revoke sessions, notify IAM owner Privileged admin reset → approval required
Data exfiltration (PHI risk) Block egress, alert compliance, preserve evidence Always require compliance + SecOps approval

Human-in-the-loop gate (healthcare)

High clinical-impact actions—such as isolating a host on a patient-care VLAN, disabling accounts for clinical staff, or blocking egress from EHR-adjacent systems—must route through a CHOA analyst approval step in Splunk SOAR before execution. Low-impact actions (block known-bad IOC, quarantine non-clinical endpoint) may auto-execute per agreed runbook.

Where feasible, sync SOAR action status back to GreyMatter so ReliaQuest has closed-loop visibility into containment progress.

5. CHOA team investigation & containment (in Splunk ES)

L2/L3 ownership in Splunk ES

  • CHOA L2/L3 analysts own the escalated case within Splunk ES with full GreyMatter context attached.
  • Use Splunk ES Investigation Workbench to pivot across correlated events and assets.
  • Coordinate containment with clinical and operational stakeholders given healthcare uptime sensitivity.
  • Document root cause and remediation in the Splunk ES case record.

Patient safety & uptime context

Healthcare containment decisions must balance security urgency against clinical continuity. Before isolating systems, disabling accounts, or blocking network paths, confirm impact with the appropriate clinical or operational owner. Splunk ES case notes should capture stakeholder coordination, approval decisions, and any deferred actions.

6. Closure, feedback & continuous improvement

Closed-loop tuning

  • Push closed-case outcomes back to GreyMatter to tune ReliaQuest detection fidelity.
  • Hold recurring joint tuning reviews to refine correlation searches, escalation thresholds, and RBA weights.
  • Track metrics: MTTD, MTTR, escalation accuracy, false-positive rate, and SLA adherence.
  • Run periodic tabletop exercises to validate the cross-platform escalation and SOAR paths.

Joint metrics dashboard

Metric Owner Purpose

MTTR (mean time to respond) CHOA Track L2/L3 containment and remediation speed

False-positive rate ReliaQuest FP closures in GreyMatter vs. total alerts triaged

Decision points

Critical success factors

What makes this model work

  • Clean data contract: Define exactly what telemetry flows into GreyMatter and what triggers escalation back.
  • Shared case ID: Maintain a single correlation ID linking GreyMatter and Splunk ES cases.
  • Closed-loop visibility: Sync SOAR/response outcomes back to GreyMatter for MSSP awareness.
  • Healthcare guardrails: PHI handling and patient-safety context shape data sharing and containment.

Implementation readiness checklist

  • Data contract signed: in-scope sources, PHI exclusions, residency rules documented
  • API connector tested: notables forward to GreyMatter with ATT&CK mapping
  • Escalation payload schema agreed: all required fields validated in test environment
  • shared_case_id reconciliation tested: GreyMatter case ↔ Splunk ES notable linkage verified
  • SOAR playbooks mapped to escalation scenarios with HITL gates for clinical systems
  • SLA timers configured per severity on both platforms
  • Closed-loop sync enabled: case closure and SOAR status return to GreyMatter
  • Joint tabletop exercise completed: end-to-end escalation and SOAR path validated
  • Metrics dashboard live: MTTD, MTTR, escalation accuracy, FP rate, SLA adherence
  • Recurring joint tuning review cadence established (monthly or quarterly)
